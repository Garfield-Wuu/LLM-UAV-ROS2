## 第一章 总体分层架构（Hierarchical Architecture）

### 1.1 六层解耦架构概览

系统采用严格的六层解耦架构设计，各层之间通过 ROS 2 话题（Topic）传递结构化消息，单一层级的替换或升级不影响其他层级的运行。

```
┌──────────────────────────────────────────────────────────────────┐
│  Layer 1: 任务理解层（Task Understanding）                        │
│  llm_client.py  →  Groq / Ollama  →  /uav/user_command (JSON)    │
├──────────────────────────────────────────────────────────────────┤
│  Layer 2: 视觉感知层（Visual Perception）                         │
│  yolo_world_detector.py  →  /uav/detections_2d                   │
├──────────────────────────────────────────────────────────────────┤
│  Layer 3: 几何 Grounding 层（Geometric Grounding）                │
│  target_grounding_node.py  →  /uav/semantic_targets_camera        │
├──────────────────────────────────────────────────────────────────┤
│  Layer 4: 状态估计层（State Estimation）                          │
│  semantic_target_tf_node.py  +  AirSim odom（→ VINS-Fusion）     │
│  →  /uav/semantic_targets_world                                   │
├──────────────────────────────────────────────────────────────────┤
│  Layer 5: 轨迹规划层（Trajectory Planning）                       │
│  text_command_bridge.py  +  EGO-Planner（可选）                   │
│  →  /hw_insight/keyboard_velocity                                 │
├──────────────────────────────────────────────────────────────────┤
│  Layer 6: 控制执行层（Control Execution）                         │
│  move_velocity.py  →  PX4 Offboard  →  AirSim / 真机             │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 各层输入输出与技术选型

| 层级 | 输入 | 输出 | 核心技术 | 选型理由 |
|------|------|------|---------|---------|
| 任务理解 | 自然语言文本 | 结构化 JSON action | Groq API / Ollama | Groq 云端低延迟（<2s），Ollama 本地隐私保护；两者均兼容 OpenAI 接口规范 |
| 视觉感知 | RGB 图像 + 文本 prompt | 2D 检测框 + 置信度 | YOLO-World-S (GPU) | 开放词汇检测，无需针对新目标重新训练；GPU 单帧 <100ms |
| 几何 Grounding | 2D bbox + 深度图 + 相机内参 | 相机坐标系 3D 点 | 针孔模型逆投影 + 深度中位数 | 中位数对遮挡和传感器噪点鲁棒；单层职责，独立可测 |
| 状态估计 | 相机系 3D 点 + 里程计位姿 | ENU world 坐标系 3D 点 | TF2 旋转矩阵链 + NED→ENU | 集中坐标转换，上下游不感知坐标系差异；后续切 VINS-Fusion 只需 remap |
| 轨迹规划 | 目标点 + 当前位置 | NED 速度 setpoint | P 控制器 + EGO-Planner（可选）| 比例控制器在 Offboard 模式下响应快、实现简单；EGO-Planner 提供梯度优化无碰撞轨迹 |
| 控制执行 | NED 速度 setpoint | PX4 Offboard 指令 | PX4 uXRCE-DDS | uXRCE-DDS 相比 MAVROS 2 延迟更低，无需 ROS 1 桥接，与 PX4 v1.14+ 原生兼容 |

### 1.3 通信中间件选型：Micro-XRCE-DDS vs. MAVROS 2

系统选择 **Micro-XRCE-DDS + px4_msgs** 替代 MAVROS 2 作为 ROS 2 与 PX4 之间的通信桥接，主要原因如下：

1. **延迟更低**：uXRCE-DDS 代理直接在 PX4 uORB 与 ROS 2 DDS 之间进行 1:1 消息映射，无需经过 MAVROS 的 MAVLink 序列化/反序列化层，端到端延迟可降低约 30%。
2. **原生 ROS 2 类型**：`px4_msgs` 直接映射 PX4 内部消息类型，`VehicleLocalPosition`、`VehicleStatus` 等字段与 PX4 文档对齐，无需字段转换。
3. **官方维护**：PX4 v1.14 以上版本将 uXRCE-DDS 作为官方推荐桥接方案，MAVROS 2 对新版 PX4 的支持滞后。
4. **QoS 细粒度控制**：安全关键通道使用 `BEST_EFFORT + TRANSIENT_LOCAL`，保证消息不因网络抖动丢失；普通状态话题使用默认 QoS 降低开销。

## 第二章 关键模块实现细节（Module Implementation）

### 2.1 任务理解层：LLM 客户端

#### 2.1.1 Prompt Engineering 与遥测上下文注入

`llm_client.py` 的系统 Prompt 由五个结构化段落组成，每次 LLM 调用前通过 `_build_messages()` 动态注入最新遥测：

```
━━━ 可用 Actions（只能从此列表选择，含 FIND_AND_GOTO 视觉任务动作）━━━
━━━ 坐标轴方向（机体坐标 vs NED 坐标的详细说明，防止 LLM 幻觉方向错误）━━━
━━━ 安全约束（最大高度 120m，最大速度 15 m/s）━━━
━━━ 当前无人机状态（从最新 TELEMETRY 注入：位置、速度、高度、航向、flight_phase）━━━
━━━ 输出规则（单步 JSON / 多步 plan / 不明确→HOVER，禁止解释文字）━━━
```

遥测注入示例：

```
当前无人机状态：
  武装: ARMED   飞行阶段: HOVERING
  位置(NED): x=8.2m  y=3.1m  高度=6.0m
  速度: vx=0.0  vy=0.0  vz=0.0 m/s
  航向: 45°   当前指令: HOVER
```

LLM 以 `temperature=0.1` 调用，接近确定性输出以减少随机幻觉。对 Groq 启用 `response_format: {type: json_object}` 强制纯 JSON 输出；对 Ollama 启用 `format: json`。

#### 2.1.2 结构化字段容错提取算法

LLM 响应可能包含 Markdown 代码围栏、`<think>` 推理块（DeepSeek-R1 等模型）或嵌套解释文字，`_extract_json_robust()` 采用三级降级策略确保鲁棒提取：

```
算法：_extract_json_robust(raw_text)

Step 1：预处理
  text ← strip_markdown_fences(raw_text)     // 移除 ``` 和 ```json 围栏
  text ← strip_think_blocks(text)            // 移除 <think>...</think>（保留推理内容日志）

Step 2：S1 直接解析
  try: return json.loads(text)
  except JSONDecodeError: continue

Step 3：S2 平衡括号候选提取
  candidates ← []
  depth ← 0, start ← -1
  for i, ch in enumerate(text):
      if ch == '{':
          if depth == 0: start = i
          depth += 1
      elif ch == '}':
          depth -= 1
          if depth == 0 and start != -1:
              candidates.append(text[start:i+1])
  candidates.sort(key=len, descending=True)   // 优先最长候选（通常含更多上下文）
  for cand in candidates:
      obj = json.loads(cand)
      if 'action' in obj or 'plan' in obj:
          return obj

Step 4：S3 正则兜底
  patterns = [
      r'\{[^{}]*"action"\s*:\s*"[^"]+?"[^{}]*\}',   // 单条指令（禁止嵌套，防止过匹配）
      r'\{\s*"plan"\s*:\s*\[[\s\S]*?\]\s*\}',        // 多步计划（惰性匹配）
  ]
  for pattern in patterns:
      m ← re.search(pattern, text, DOTALL)
      if m: return json.loads(m.group())

return None   // 三级均失败，上层回退到 HOVER
```

`<think>` 块的完整内容被持久化到 ROS 2 日志文件（`~/.ros/log/`），终端仅显示首行摘要，保留推理可解释性而不干扰操作流程。

#### 2.1.3 安全层校验

提取到 JSON 后，系统依次执行：

1. **别名归一化**：将模型幻觉产生的同义动词（如 `LIFTOFF → TAKEOFF`，`SEEK → FIND_AND_GOTO`）映射到标准 action 名，共维护 40+ 条别名规则（`_ACTION_ALIASES`）。
2. **参数类型强制转换**：`_coerce_params()` 按 `_PARAM_SCHEMA` 中定义的 `(type_fn, min, max, required)` 四元组对参数进行强制类型转换和下界截断。
3. **安全边界截断**：高度上限 120 m，水平速度上限 15 m/s，垂直速度上限 5 m/s，偏航角速度上限 1.5 rad/s。
4. **地面状态自动起飞注入**：若 `flight_phase == 'GROUND'` 且收到飞行动作，系统自动在计划首步插入 `TAKEOFF`，防止飞机在地面执行移动指令。
5. **`FIND_AND_GOTO` 特殊处理**：`query` 为空字符串时回退到 `HOVER`；`query` 非空时直接透传，不做高度/速度截断（该 action 无对应参数）。

### 2.2 视觉感知与几何 Grounding 层

#### 2.2.1 YOLO-World 开放词汇检测流程

YOLO-World 采用"提示-检测"（prompt-then-detect）范式：文本编码器（CLIP ViT-B/32，已完全离线化）将用户提供的自然语言描述编码为文本特征向量，在推理时与图像特征进行跨模态匹配。这使得系统无需针对特定目标类别重新训练，仅通过修改文本 prompt 即可实现开放词汇检测。

`yolo_world_detector.py` 在独立推理线程中运行，通过 `min_inference_interval_sec` 参数限制最高检测频率，避免 GPU 过载。图像以临时文件形式写入 `/tmp/` 再由 mmdet pipeline 读取，该设计规避了对 mmdet 上游代码的修改，`/tmp/` 写入开销在 GPU 推理耗时（<100ms）面前可忽略。

检测结果以如下 JSON 格式发布至 `/uav/detections_2d`：

```json
{
  "prompt": "person wearing yellow clothes",
  "image_stamp": 1774123456.789,
  "detections": [
    {
      "label": "person",
      "score": 0.72,
      "bbox": {"x_min": 120, "y_min": 80, "x_max": 280, "y_max": 420},
      "stamp": 1774123456.789,
      "frame_id": "camera_optical"
    }
  ]
}
```

#### 2.2.2 深度逆投影数学实现

`target_grounding_node.py` 实现从像素坐标到相机系三维点的完整变换链：

**Step 1：边界框中位数深度估计**

对检测框内像素点按步长 `stride`（默认 4）进行采样，对有效深度值（满足 $d_{min} < d < d_{max}$ 且 `isfinite`）取中位数作为目标深度估计：

$$
d_{target} = \mathrm{median}\left\{ d_{ij} \mid (i,j) \in \Omega_{bbox},\ d_{min} < d_{ij} < d_{max},\ d_{ij} \in \mathbb{R} \right\}
$$

中位数相比均值对遮挡、深度跳变和传感器噪点具有更强的鲁棒性（breakdown point 为 50%）。

**Step 2：像素坐标 → 相机光学坐标系逆投影**

设检测框中心像素坐标为 $(u, v)$，相机内参为焦距 $(f_x, f_y)$ 和主点 $(c_x, c_y)$，目标深度为 $d$（DepthPlanar 类型，即相机平面法向量方向的深度），则相机光学坐标系（ROS 约定：x 右、y 下、z 前）下的三维点为：

$$
\begin{pmatrix} x_{cam} \\ y_{cam} \\ z_{cam} \end{pmatrix} = \begin{pmatrix} \dfrac{(u - c_x) \cdot d}{f_x} \\[6pt] \dfrac{(v - c_y) \cdot d}{f_y} \\[6pt] d \end{pmatrix}
$$

该公式由针孔相机模型（Pinhole Camera Model）的投影方程直接反推得到，适用于 AirSim `DepthPlanar` 深度类型。注意：`DepthPlanar`（平面深度）与 `DepthPerspective`（射线深度）在计算时使用不同公式，两者不可混用。

#### 2.2.3 相机坐标系 → 世界坐标系变换链

`semantic_target_tf_node.py` 实现完整的三层坐标变换，从相机光学坐标系最终输出 ENU 世界坐标：

**完整变换链**：

$$
\mathbf{p}_{world} = \mathbf{R}_{body \to world} \cdot \left( \mathbf{R}_{cam \to body} \cdot \mathbf{p}_{cam} + \mathbf{t}_{cam \to body} \right) + \mathbf{t}_{body \to world}
$$

**（1）相机光学帧 → 机体 NED 帧**（默认正前方安装，无倾斜）：

$$
\mathbf{R}_{cam \to body} = \begin{pmatrix} 0 & 0 & 1 \\ 1 & 0 & 0 \\ 0 & 1 & 0 \end{pmatrix}
$$

即：$x_{body}^{forward} = z_{cam}^{forward}$，$y_{body}^{right} = x_{cam}^{right}$，$z_{body}^{down} = y_{cam}^{down}$。

**（2）机体帧 → 里程计帧**（姿态旋转，由四元数计算）：

由里程计四元数 $\mathbf{q} = [w, x, y, z]$ 转换为旋转矩阵：

$$
\mathbf{R}(\mathbf{q}) = \begin{pmatrix} 1-2(y^2+z^2) & 2(xy-zw) & 2(xz+yw) \\ 2(xy+zw) & 1-2(x^2+z^2) & 2(yz-xw) \\ 2(xz-yw) & 2(yz+xw) & 1-2(x^2+y^2) \end{pmatrix}
$$

**（3）NED 里程计帧 → ENU 世界帧**（固定坐标系转换）：

$$
\mathbf{R}_{NED \to ENU} = \begin{pmatrix} 0 & 1 & 0 \\ 1 & 0 & 0 \\ 0 & 0 & -1 \end{pmatrix}
$$

即：$x_{ENU}^{East} = y_{NED}^{East}$，$y_{ENU}^{North} = x_{NED}^{North}$，$z_{ENU}^{Up} = -z_{NED}^{Down}$。

最终输出的 `/uav/semantic_targets_world` 中，`x_world` 为东向，`y_world` 为北向，`z_world` 为高度（朝上为正）。

### 2.3 规划与控制层

#### 2.3.1 比例导航控制器

`text_command_bridge.py` 内置一个运行于 **10 Hz** 控制环的比例导航控制器，支持四种持续运动控制模式：

**GOTO_NED 控制律**：

$$
v_{h} = \min\left(v_{max},\ K_P \cdot d_{xy}\right), \quad d_{xy} = \sqrt{(x_t - x)^2 + (y_t - y)^2}
$$

$$
\begin{pmatrix} v_x \\ v_y \end{pmatrix} = \frac{v_h}{d_{xy}} \begin{pmatrix} x_t - x \\ y_t - y \end{pmatrix}, \quad v_z = \mathrm{clip}\left(K_P \cdot (z_t - z),\ -v_{z,max},\ v_{z,max}\right)
$$

其中 $K_P = 0.8$，水平速度上限 $v_{max} = 15$ m/s，到达判据为三维距离 $d_{3D} \leq 0.5$ m。

**ORBIT 轨迹合成**：

$$
\theta_{t+1} = \theta_t + \omega \cdot \Delta t, \quad \omega = \frac{v_{tangential}}{r}
$$

$$
\mathbf{v}_{tangential} = \begin{pmatrix} -v \sin\theta \\ v \cos\theta \end{pmatrix}, \quad \mathbf{v}_{correction} = K_{orbit} \begin{pmatrix} x_{desired} - x \\ y_{desired} - y \end{pmatrix}
$$

$$
\mathbf{v}_{cmd} = \mathrm{clip}\left( \mathbf{v}_{tangential} + \mathbf{v}_{correction},\ -v_{h,max},\ v_{h,max} \right)
$$

其中 $K_{orbit} = 1.0$，初始相位角由 $\theta_0 = \mathrm{atan2}(y-c_y,\ x-c_x)$ 确定，实现无缝切入。

**MOVE_VELOCITY 机体系 → NED 旋转**：

用户指令中 `vx/vy` 以机头方向为前，通过当前航向角 $\psi$ 旋转至 NED 世界坐标系：

$$
\begin{pmatrix} v_x^{NED} \\ v_y^{NED} \end{pmatrix} = \begin{pmatrix} \cos\psi & -\sin\psi \\ \sin\psi & \cos\psi \end{pmatrix} \begin{pmatrix} v_x^{body} \\ v_y^{body} \end{pmatrix}
$$

**YAW_TO 偏航控制律**：

$$
\dot{\psi}_{cmd} = \mathrm{clip}\left(K_{yaw} \cdot \Delta\psi,\ -\dot{\psi}_{max},\ \dot{\psi}_{max}\right)
$$

其中 $\Delta\psi = \mathrm{atan2}(\sin(\psi_t - \psi),\ \cos(\psi_t - \psi)) \in [-\pi, \pi]$ 保证最短旋转方向，$K_{yaw} = 1.5$ rad/s/rad，接受阈值 $|\Delta\psi| \leq 0.05$ rad（约 3°）。

#### 2.3.2 FIND_AND_GOTO 视觉搜索状态机

系统通过以下有限状态机（FSM）实现"感知-决策-执行"的视觉任务闭环：

```
                        ┌─────────────────────────────────────────┐
                        │ 状态转换图（FIND_AND_GOTO FSM）           │
                        └─────────────────────────────────────────┘

  任何状态 ──FIND_AND_GOTO(query)──────────────────→  SEARCHING
                                                          │
                              score ≥ 0.15 目标到达       │  20s 超时
                                    ↓                     │     ↓
                              GOTO_NED ←─────────────────┘   HOVER
                                    │                         + SEARCH_FAILED 事件
                                    │ d_3D ≤ 0.5m
                                    ↓
                               HOVER（到达）
```

在 SEARCHING 状态下的完整处理流程：
1. 发布 `query` 至 `/uav/target_query`，触发 YOLO-World GPU 检测流水线
2. 无人机保持悬停（`active_command = HOVER`），搜索不影响当前高度
3. 每帧 `/uav/semantic_targets_world` 到达时，过滤置信度 ≥ 0.15 的目标，取最高分
4. 执行 ENU→NED 坐标转换：`x_ned = y_enu`（北），`y_ned = x_enu`（东）
5. 高度策略：使用无人机当前高度（而非目标物体高度），防止飞入地面
6. 调用内部 `apply_command('GOTO_NED', ...)` 完成状态切换并发布 `SEARCH_FOUND` 事件

#### 2.3.3 安全关键指令短路通道

`RTL` 和 `EMERGENCY_STOP` 绕过速度层，直接通过 `px4_msgs/VehicleCommand` 消息发布至 `/fmu/in/vehicle_command`，使用 `BEST_EFFORT + TRANSIENT_LOCAL` QoS 确保消息不因 DDS 发现延迟而丢失。该设计遵循"安全指令最短路径"原则，消除因中间节点失效而导致指令阻塞的风险。

#### 2.3.4 TAKEOFF ARM 阶段设计

起飞采用双阶段控制：

- **ARM 阶段（0 ~ 2.5s）**：持续发送 $z = -6.0$ m/s（NED 上升），触发 `move_velocity` 内部 $z < -5$ m/s 阈值，完成解锁（ARM）并切换至 Offboard 模式。2.5s 设计余量用于抵消 WSL2 时间同步抖动期间 PX4 拒绝解锁的问题。
- **高度保持阶段（2.5s 后）**：切换为 P 控制器，增益 $K_P = 0.8$，最大垂直速度 3.0 m/s，到达目标高度 ±0.35 m 范围后输出零速悬停。

## 第三章 通信协议与数据流（Communication & Data Flow）

### 3.1 核心 ROS 2 Topic 映射表

| Topic 名称 | 消息类型 | 发布频率 | 方向 | 作用 |
|-----------|---------|---------|------|------|
| `/uav/user_command` | `std_msgs/String` (JSON) | 按需 | llm_client → bridge | LLM 输出的结构化任务原语 |
| `/uav/nl_input` | `std_msgs/String` | 按需 | 外部 → llm_client | 远程/程序化自然语言输入 |
| `/uav/llm_task_status` | `std_msgs/String` (JSON) | 5 Hz + 事件 | bridge → llm_client | 遥测心跳 + 执行事件（RECEIVED / MAPPED / PUBLISHED / SEARCH_FOUND / SEARCH_FAILED）|
| `/uav/target_query` | `std_msgs/String` | 按需 | bridge → detector | FIND_AND_GOTO 触发后的视觉搜索 prompt |
| `/uav/detections_2d` | `std_msgs/String` (JSON) | ~10 Hz | yolo_world → grounding | YOLO-World 2D 检测框 + 置信度 |
| `/uav/semantic_targets_camera` | `std_msgs/String` (JSON) | ~10 Hz | grounding → tf_node | 相机坐标系 3D 目标点（含深度）|
| `/uav/semantic_targets_world` | `std_msgs/String` (JSON) | ~10 Hz | tf_node → bridge | ENU world 坐标系目标点 |
| `/uav/target_goal` | `geometry_msgs/PoseStamped` | ~10 Hz | tf_node / bridge | EGO-Planner 目标点（ENU frame）|
| `/uav/semantic_target_marker` | `visualization_msgs/Marker` | ~10 Hz | tf_node → RViz | 目标点可视化（红色球体，TTL=3s）|
| `/hw_insight/keyboard_velocity` | `hw_interface/HWSimpleKeyboardInfo` | 10 Hz | bridge → move_velocity | 速度 setpoint（NED 世界坐标系）|
| `/fmu/out/vehicle_local_position` | `px4_msgs/VehicleLocalPosition` | ~50 Hz | PX4 → bridge | NED 位置、速度、航向（弧度）|
| `/fmu/out/vehicle_status` | `px4_msgs/VehicleStatus` | ~10 Hz | PX4 → bridge | 武装状态（arming_state）、导航状态（nav_state）|
| `/fmu/in/vehicle_command` | `px4_msgs/VehicleCommand` | 按需 | bridge → PX4 | RTL / EMERGENCY_STOP 直发通道 |
| `/airsim_node/PX4/CameraDepth1/Scene` | `sensor_msgs/Image` (BGR8) | ~30 Hz | AirSim → detector | RGB 图像 |
| `/airsim_node/PX4/CameraDepth1/DepthPlanar` | `sensor_msgs/Image` (32FC1) | ~30 Hz | AirSim → grounding | 深度图（单位：米）|

### 3.2 遥测闭环：Telemetry 如何影响 LLM 决策

`text_command_bridge` 以 5 Hz 频率持续发布 TELEMETRY 消息，`llm_client` 订阅后缓存最新状态。每次 LLM 推理调用前，`_build_messages()` 将最新遥测注入系统 Prompt，形成以下闭环：

```
LLM 输出 JSON ──→ /uav/user_command ──→ bridge 执行 ──→ PX4 物理响应
       ↑                                                        │
       │      遥测注入 Prompt ←── /uav/llm_task_status ←────────┘
       │   (位置、速度、高度、航向、武装状态、flight_phase、当前指令)
       │
       └──── 用户输入新指令（stdin 或 /uav/nl_input）
```

**遥测新鲜度守卫**：`_TELEMETRY_STALE_SEC = 3.0`。若超过 3 秒未收到新 TELEMETRY，`_system_online()` 返回 `False`，所有普通飞行指令被安全层拦截，终端打印"系统离线"警告。`RTL`/`LAND`/`HOVER`/`EMERGENCY_STOP` 四类安全指令豁免，确保任何情况下均可触发安全行为。

**FIND_AND_GOTO 搜索进度透传**：搜索进行中时，TELEMETRY 额外携带 `searching_query` 和 `search_remaining_sec` 字段，使 LLM 在下次交互时感知搜索状态，避免重复触发搜索任务。

## 第四章 核心算法与数学表述（Core Algorithms & Mathematics）

### 4.1 像素到 ENU 世界坐标的端到端变换

从相机像素坐标 $(u, v)$ 到 ENU 世界坐标 $\mathbf{p}_{world}$ 的端到端变换：

$$
\mathbf{p}_{world} = \mathbf{R}_{NED \to ENU} \cdot \mathbf{R}(\mathbf{q}) \cdot \left( \mathbf{R}_{cam \to body} \cdot \begin{pmatrix} \frac{(u-c_x)d}{f_x} \\ \frac{(v-c_y)d}{f_y} \\ d \end{pmatrix} + \mathbf{t}_{cam} \right) + \mathbf{R}_{NED \to ENU} \cdot \mathbf{p}_{odom}
$$

各矩阵的数值定义已在 §2.2.3 给出。

### 4.2 GOTO_NED 速度平滑性分析

比例控制策略实现速度随距离自动平滑衰减，在接近目标时无需额外减速规划：

$$
v(t) = \min\left(v_{max},\ K_P \cdot d(t)\right)
$$

当距离较大时速度饱和于 $v_{max}$，接近目标时比例衰减，自然形成"加速-巡航-减速"三段式轨迹。到达判据 $d_{3D} \leq 0.5$ m 匹配 PX4 速度控制模式的典型位置估计漂移余量。

### 4.3 ORBIT 轨道误差收敛特性

径向跟踪误差 $e_r = r_{actual} - r_{desired}$ 在向心修正速度的作用下以一阶指数规律收敛：

$$
\dot{e}_r \approx -K_{orbit} \cdot e_r \implies e_r(t) = e_r(0) \cdot e^{-K_{orbit} \cdot t}
$$

时间常数 $\tau = 1/K_{orbit} = 1.0$ s，理论上约 3 s（$3\tau$）内径向误差降至初始误差的 5% 以内。

### 4.4 JSON 容错提取中的正则表达式

S3 策略使用的正则模式（DOTALL 模式，允许点匹配换行符）：

```regex
单步指令：\{[^{}]*"action"\s*:\s*"[^"]+?"[^{}]*\}
多步计划：\{\s*"plan"\s*:\s*\[[\s\S]*?\]\s*\}
```

前者使用 `[^{}]*` 限定单层括号，防止跨越嵌套结构导致的过匹配；后者使用 `[\s\S]*?` 惰性匹配，允许 plan 数组包含任意数量的步骤对象。

### 4.5 多步计划执行等待策略

顺序计划执行中，`_wait_command_done()` 采用差异化等待策略以平衡确定性与响应性：

| 动作类型 | 等待策略 | 超时 |
|---------|---------|------|
| `SET_SPEED`, `EMERGENCY_STOP` | 立即返回（无等待） | — |
| `MOVE_VELOCITY`, `MOVE_REL`, `HOVER`, `ORBIT`（有 duration）| 等待 `duration + 1.2s`（稳定缓冲）| — |
| `RTL`, `LAND` | 轮询 `arming_state == DISARMED`（落地判据）| 60s |
| `TAKEOFF` | 轮询高度：`|alt - target_alt| < 0.6m` | 20s |
| `GOTO_NED`, `YAW_TO`, `ORBIT`（无 duration）| 轮询 TELEMETRY：`command ∈ {IDLE, HOVER}` | 30s |
